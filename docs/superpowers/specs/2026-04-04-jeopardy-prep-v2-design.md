# Jeopardy Prep v2 — Design Spec

## Context

The original Jeopardy Prep app was built a year ago as a simple Next.js app using OpenAI GPT-4 for question generation. It has no persistence, no tests, a stubbed-out J-Archive integration, and a client-side exposed API key. This is a full rethink: new platform (Lovable), new AI backend (Claude), real Jeopardy data, cloud persistence, multi-user support, and two distinct practice modes.

## Platform

**Lovable** (AI-powered app builder) — generates React + Supabase apps. The output of this spec is intended to be fed directly to Lovable for implementation.

**Stack:**
- Frontend: React (generated by Lovable)
- Backend: Supabase (Postgres, Auth, Edge Functions)
- AI: Claude API (via Supabase Edge Functions)
- Question Data: jwolle1/jeopardy_clue_dataset (538K+ real Jeopardy clues)

## Game Modes

### 1. Game Mode — Simulated Jeopardy

A visual board-based game simulating the Jeopardy experience.

**Board:**
- 6 categories x 5 clues ($200, $400, $600, $800, $1000)
- Questions sourced from the question bank (Supabase `questions` table)
- User clicks a cell to reveal the clue in a modal
- User types answer in "What is..." format
- 1 Daily Double per game (randomly placed) — user wagers points before seeing the clue
  - Wager rules: minimum $5, maximum is user's current score (or $1000 if score is below $1000)

**Final Jeopardy:**
- Single category revealed after the board is cleared or user clicks "End Round"
- User wagers from their current score
- One clue, one answer

**Scoring:**
- Correct: +clue value (or wager for Daily Double / Final Jeopardy)
- Incorrect: -clue value (or wager)
- Partial credit (via Claude evaluation): +half value

**End of game:**
- Summary screen: total score, accuracy %, category breakdown
- Session saved to database

### 2. Drill Mode — Adaptive Practice

Rapid-fire clue practice that adapts to the user's weaknesses.

**Flow:**
- No board — single clue card at a time
- User can pick specific topics or let the system choose based on weak areas
- Answer → immediate feedback → next clue

**Adaptive logic:**
- Checks `category_stats` table for categories where `correct_count / total_attempts < 60%`
- Prioritizes weak categories in question selection
- If the question bank is thin for a weak category, calls Claude to generate targeted questions

**Spaced repetition:**
- Missed questions are resurfaced after intervals: 1 day, 3 days, 7 days
- Tracked via `game_answers.next_review_at` field (set on incorrect/partial answers)
- When selecting drill questions, prioritize those with `next_review_at <= now()`

## Answer Evaluation (Hybrid)

### Fast Path (Local)
1. Normalize both answers: strip "What/Who/Where is/are...", lowercase, remove punctuation
2. Extract keywords (words 3+ characters)
3. If **80%+ keyword match** → correct (award full points)
4. If **0% match** → incorrect (deduct points)

### Claude Path (Borderline)
5. For matches between **1-79%**, call the `evaluate-answer` Edge Function
6. Claude receives: correct answer, user's answer
7. Claude returns: correct / partially correct / incorrect + brief explanation
8. Partial credit: half points for "partially correct"
9. Explanation shown to user as feedback

## Data Model (Supabase)

### Tables

**profiles**
- `id` (uuid, FK to auth.users)
- `display_name` (text)
- `preferred_categories` (text[])
- `daily_goal` (integer, default 20)
- `created_at` (timestamptz)

**questions**
- `id` (uuid, PK)
- `category` (text, indexed)
- `difficulty` (integer: 200, 400, 600, 800, 1000)
- `clue` (text)
- `answer` (text)
- `source` (text: 'jarchive' | 'claude')
- `verified` (boolean, default true for jarchive, false for claude)
- `air_date` (date, nullable)
- `round` (text: 'jeopardy' | 'double_jeopardy' | 'final_jeopardy', nullable)
- `created_at` (timestamptz)

**game_sessions**
- `id` (uuid, PK)
- `user_id` (uuid, FK to auth.users)
- `mode` (text: 'game' | 'drill')
- `score` (integer)
- `total_possible` (integer)
- `started_at` (timestamptz)
- `completed_at` (timestamptz)

**game_answers**
- `id` (uuid, PK)
- `session_id` (uuid, FK to game_sessions)
- `question_id` (uuid, FK to questions)
- `user_answer` (text)
- `is_correct` (boolean)
- `is_partial` (boolean, default false)
- `points_earned` (integer)
- `evaluation_method` (text: 'local' | 'claude')
- `answered_at` (timestamptz)
- `next_review_at` (timestamptz, nullable — set for incorrect/partial answers in drill mode)

**category_stats**
- `id` (uuid, PK)
- `user_id` (uuid, FK to auth.users)
- `category` (text)
- `total_attempts` (integer, default 0)
- `correct_count` (integer, default 0)
- `last_attempted` (timestamptz)
- UNIQUE constraint on (user_id, category)

### Row Level Security
- All tables: users can only read/write their own data (filtered by `user_id = auth.uid()`)
- `questions` table: read access for all authenticated users, insert for Edge Functions only

## Supabase Edge Functions

### 1. `generate-questions`
- **Trigger:** Drill mode needs questions for a category not well-covered in the bank
- **Input:** `{ category: string, difficulty: number, count: number }`
- **Action:** Calls Claude API with a prompt to generate Jeopardy-style clues
- **Claude prompt:** Instructs Claude to generate clues in authentic Jeopardy format — a statement (clue) where the correct response is in "What is...?" form. Specifies category, difficulty level, and asks for JSON output.
- **Output:** Inserts generated questions into `questions` table (source: 'claude', verified: false), returns them to the client
- **Claude model:** claude-sonnet-4-6 (fast, cost-effective for generation)

### 2. `evaluate-answer`
- **Trigger:** Local keyword match scores between 1-79%
- **Input:** `{ correct_answer: string, user_answer: string, clue: string }`
- **Action:** Calls Claude API to judge the answer
- **Claude prompt:** "Given this Jeopardy clue and the correct answer, evaluate the user's response. Return JSON: { verdict: 'correct' | 'partial' | 'incorrect', explanation: string }"
- **Output:** Returns verdict and explanation to the client
- **Claude model:** claude-haiku-4-5 (fastest, cheapest — evaluation is a simple task)

## Question Bank Seeding

**Source:** [jwolle1/jeopardy_clue_dataset](https://github.com/jwolle1/jeopardy_clue_dataset) — 538,845 clues, TSV format.

**Seeding process:**
1. Download the TSV dataset
2. Write a seed script that parses the TSV and maps fields to the `questions` table schema:
   - `round` → map to 'jeopardy' / 'double_jeopardy' / 'final_jeopardy'
   - `clue_value` → `difficulty`
   - `clue` → `clue`
   - `answer` → `answer` (strip HTML tags if present)
   - `category` → `category`
   - `air_date` → `air_date`
   - `source` → 'jarchive'
   - `verified` → true
3. Run the seed script against the Supabase database (can use Supabase CLI or direct SQL import)
4. Create indexes on `category` and `difficulty` for fast querying

## Verification Plan

### Functional Testing
1. **Auth flow:** Sign up, log in, log out, profile creation
2. **Game mode:** Start game → board renders with 6 categories → click clue → answer → score updates → Daily Double wager → Final Jeopardy → summary screen → session saved to DB
3. **Drill mode:** Start drill → clue appears → answer → feedback → next clue → weak category prioritization works → session saved
4. **Answer evaluation:** Test all three paths — clear correct (80%+), clear incorrect (0%), borderline (1-79% triggers Claude)
5. **Question bank:** Verify seeded data loads correctly, categories query is fast, Claude generation works for missing categories

### Edge Cases
- Empty categories (no questions for a category in the bank)
- Daily Double on first clue (score is 0, wager minimum is $5)
- User closes mid-game (session should be saved as incomplete)
- Claude API timeout or error (graceful fallback: accept local match result)

### Performance
- Board query should return in < 500ms (indexed category + difficulty)
- Claude evaluation should respond in < 2s
- 538K questions should not cause storage or query issues on Supabase free tier
